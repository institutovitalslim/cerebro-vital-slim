Resumo executivo

O Onlook é relevante para nós porque ataca uma dor real: acelerar criação/edição de interfaces React/Next.js com uma camada visual, mantendo vínculo com código real. Para o IVS, isso encaixa especialmente em landing pages, funis, páginas da Clara, painéis internos, relatórios HTML e aplicação consistente do design system IVS.

Minha recomendação: não instalar como ferramenta central agora. Usar como referência para criar um IVS Visual Layer menor e governado: editor/gerador visual de páginas IVS-first, com templates próprios, tokens de marca, pré-visualização, diffs e aprovação antes de publicar.

O que o repo faz

  • Cria apps Next.js rapidamente a partir de texto, imagem ou templates.
  • Edita visualmente a UI em navegador/DOM com experiência tipo Figma.
  • Permite preview em tempo real.
  • Gerencia assets, tokens de marca, páginas e camadas.
  • Integra AI para gerar e aplicar alterações.
  • Tem branching/checkpoints para experimentar designs.
  • Usa script preload para comunicação direta com DOM do app renderizado.

Arquitetura observada

  • apps/web/client: front-end Next.js do editor.
  • apps/web/server: servidor de controle com Fastify/WebSocket/tRPC.
  • apps/web/preload: script injetado no app/template para ler e manipular DOM via iframe.
  • apps/backend: Supabase local/backend.
  • packages/ai: agentes, chat, tools, prompts e streaming.
  • packages/parser: edição de código, estilo, layout, texto, imagens e transformação.
  • packages/code-provider: abstração para CodeSandbox e filesystem local.

O que aproveitar

  • Preload/iframe DOM bridge: padrão muito útil para selecionar elementos visualmente.
  • Parser/aplicador de mudanças: separar intenção visual de patch em código.
  • Code provider abstraction: trocar backend de código sem acoplar tudo.
  • Checkpoints/branching: indispensável para não quebrar páginas em produção.
  • Tokens/assets/layers: base para design system IVS operacional.

O que descartar

  • Não trazer a plataforma inteira para produção IVS.
  • Não expor projetos com dados sensíveis/pacientes/leads dentro de editor visual genérico.
  • Não depender de CodeSandbox/Supabase externo para páginas críticas.
  • Não permitir “AI apply” direto em produção sem diff, preview e aprovação.
  • Não transformar em ferramenta aberta para equipe sem permissões e trilha de auditoria.

Plano recomendado — IVS Visual Layer

  1. MVP 1: editor visual local para páginas HTML/Next do IVS com seleção de elemento + edição de texto/classes Tailwind.
  2. MVP 2: biblioteca de componentes/tokens IVS: hero, CTA, formulário Typeform-like, cards, provas sociais, seção médica/compliance.
  3. MVP 3: AI assistida, mas governada: prompt → diff → preview → aprovação.
  4. MVP 4: export para landing/funil com checklist LGPD/compliance e bloqueio de PII.
  5. MVP 5: integração com fluxo do João/Clara para páginas de campanha, sem publicação automática.

Decisão final

Adaptar como referência. O Onlook é uma excelente inspiração para acelerar front-end/design no IVS, mas a melhor estratégia é criar uma versão menor, própria, governada e conectada ao nosso design system — não instalar o repo inteiro na operação.

Próximo passo sugerido: se Tiaro aprovar, eu preparo uma spec curta do ivs-visual-layer e um MVP local com uma página IVS real/sintética para provar: selecionar elemento → editar → gerar diff → preview.
