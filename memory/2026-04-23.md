### Bridge HTTPS para o OpenClaw
- Problema: Claude no sandbox precisava monitorar gateway `localhost:18789` da VPS sem SSH.
- Solução: nginx + Let's Encrypt + bearer auth, subdomínio próprio. DNS na HostGator, A record `openclaw.institutovitalslim.com.br` → `187.77.58.193`.
- Scripts: `ops/setup/openclaw-bridge.sh` (install idempotente) e `ops/setup/rotate-bridge-token.sh`.
- **Princípio do menor privilégio**: nginx valida `BRIDGE_TOKEN` na borda mas **strippa** o header `Authorization` antes de proxy. Endpoints que exigem `gateway.auth.token` interno devolvem 401 via bridge — intencional. Vazamento do BRIDGE_TOKEN expõe `/health`, `/version`, UI pública; não dispara skills nem mexe em paciente.
- Token rotacionado após exposição em chat; antigo confirmado revogado (401).
- URL: `https://openclaw.institutovitalslim.com.br` — liveness em `/__bridge_alive`.

### Gotchas de setup do bridge
1. **Ubuntu 25.10 apt silent failure**: `apt-get install -y -qq nginx ... >/dev/null` retornou 0 sem instalar. `nginx-core` direto funcionou.
2. **hpanel browser terminal PATH restrito**: shell sem `/sbin` nem `/usr/sbin`, mesmo como root. Script exporta PATH defensivamente.
3. **nginx `map_hash_bucket_size` default = 64** insuficiente para "Bearer <64 hex>" (71 chars). Setado 128.
4. **iOS Smart Punctuation**: URLs digitadas no terminal viram `<URL>` (bash interpreta como redirect). Evitar URLs em CLI quando possível, usar git.
5. **0x0.st desabilitado** por abuso de bots IA. Fallback funcional: `paste.rs`.
